Introduction

Aims

The file-handling chapters of this course teach you to read and write records in sequential, relative, and indexed files. That covers the disk-as-data-store role of COBOL, but it leaves out the most common backing store in commercial COBOL: a relational database, accessed via embedded SQL.

If you take a job maintaining COBOL at a bank, an insurance company, or a government agency, you will see EXEC SQL ... END-EXEC blocks on day one. This unit gives you the model for reading and writing them.

Objectives

By the end of this unit you should —

  1. Understand what the SQL precompiler does, and how host variables bridge COBOL data items and SQL placeholders.

  2. Be able to write singleton SQL statements (SELECT INTO, INSERT, UPDATE, DELETE) embedded in COBOL.

  3. Be able to write a cursor loop to iterate over a multi-row result set.

  4. Know to check SQLCODE after every call, and recognise the common return codes.

  5. Understand the role of BIND, plans, and packages in a DB2 deployment.

Prerequisites

  • The structure of COBOL programs
  • Declaring data in COBOL
  • Basic Procedure Division commands
  • Selection in COBOL
  • Iteration in COBOL
  • Basic familiarity with SQL (CREATE TABLE, SELECT, INSERT, UPDATE, DELETE).

What embedded SQL is

The precompile step

COBOL has no native notion of SQL. The way SQL gets into a COBOL source file is by way of a precompiler — a separate tool that runs before the COBOL compiler.

The precompiler reads your .cob source, finds every EXEC SQL ... END-EXEC block, and rewrites them into plain COBOL CALLs to the database client library. It also adds the data declarations the database calls need (the SQLCA, indicator variables, …). The resulting plain-COBOL file is what the COBOL compiler sees.

So a typical DB2-on-z/OS compile flow has four steps, not two:

  1. DSNHPC — the DB2 precompiler. Reads source, emits DBRM + processed source.
  2. The COBOL compiler. Reads the processed source, emits an object module.
  3. The linkage editor. Combines object module with DB2 stubs to produce a load module.
  4. BIND — binds the DBRM into a DB2 package or plan, choosing access paths.

Step 4 is the unusual one — the database holds a stored, optimised plan for your program's SQL, and rebinding can change performance even when no source code changed.

Host variables

SQL needs to read values from and write values into your COBOL program. The bridge is a host variable — an ordinary COBOL data item, declared inside a special BEGIN/END DECLARE SECTION so the precompiler knows about it, and referenced inside SQL with a leading colon. 

DATA DIVISION.
WORKING-STORAGE SECTION.

EXEC SQL BEGIN DECLARE SECTION END-EXEC.
01  WS-EMP-ID            PIC S9(9) COMP.
01  WS-EMP-NAME          PIC X(40).
01  WS-EMP-SALARY        PIC S9(7)V99 COMP-3.
EXEC SQL END DECLARE SECTION END-EXEC.

And the SQL that uses them:

EXEC SQL
    SELECT EMPNAME, SALARY
      INTO :WS-EMP-NAME, :WS-EMP-SALARY
      FROM EMPLOYEE
     WHERE EMPID = :WS-EMP-ID
END-EXEC.

The colon prefix is what tells the precompiler "this is a COBOL data name, not an SQL identifier." Without the colon, EMPID = WS-EMP-ID would try to match a column called WS-EMP-ID — not what you want.

The SQLCA

Every SQL call returns its outcome through the SQL Communication Area — a record with fields for the return code, the number of rows affected, and a textual error message. The precompiler inserts the declaration for you: 

EXEC SQL INCLUDE SQLCA END-EXEC.

The two fields you check after almost every call:

  • SQLCODE — the return code. 0 is success; positive values are warnings; negative values are errors. +100 means "no rows found / end of cursor."
  • SQLERRD(3) — the number of rows affected by the previous INSERT/UPDATE/DELETE.

EXEC SQL syntax

Singleton SELECT INTO

When you know the query will return at most one row, use SELECT INTO — the result columns go straight into host variables, no cursor needed. 

EXEC SQL
    SELECT EMPNAME, SALARY
      INTO :WS-EMP-NAME, :WS-EMP-SALARY
      FROM EMPLOYEE
     WHERE EMPID = :WS-EMP-ID
END-EXEC.

IF SQLCODE = 0
    DISPLAY "Employee: " WS-EMP-NAME " salary " WS-EMP-SALARY
ELSE IF SQLCODE = 100
    DISPLAY "No employee with that ID"
ELSE
    DISPLAY "DB error: " SQLCODE
END-IF.

If the query returns more than one row, you get SQLCODE = -811 — "row not unique." If that's actually possible, you need a cursor (see Part 3).

INSERT

EXEC SQL
    INSERT INTO EMPLOYEE (EMPID, EMPNAME, SALARY)
    VALUES (:WS-EMP-ID, :WS-EMP-NAME, :WS-EMP-SALARY)
END-EXEC.

IF SQLCODE NOT = 0
    DISPLAY "Insert failed: " SQLCODE
END-IF.

SQLERRD(3) will be 1 on success. The two non-zero codes you will see most often:

  • -803 — duplicate key.
  • -530 — foreign-key violation (parent row doesn't exist).

UPDATE and DELETE

EXEC SQL
    UPDATE EMPLOYEE
       SET SALARY = SALARY * 1.05
     WHERE DEPT = :WS-DEPT
END-EXEC.

DISPLAY SQLERRD(3) " rows updated."
EXEC SQL
    DELETE FROM EMPLOYEE
     WHERE EMPID = :WS-EMP-ID
END-EXEC.

IF SQLERRD(3) = 0
    DISPLAY "No matching row."
END-IF.

For both verbs, SQLCODE = 0 with SQLERRD(3) = 0 means "the SQL ran fine, but no rows matched the WHERE clause." That is usually a logic problem in the caller, so check for it.

Cursors

Why cursors

🔍

Notice the shape: OPEN / FETCH in a loop until end-of-data / CLOSE. It is the same pattern you already know from Processing Sequential files — OPEN INPUT / READ ... AT END / CLOSE.

A cursor is the SQL equivalent of opening a sequential file: it represents a result set you can iterate over a row at a time. The four steps are DECLARE, OPEN, FETCH, CLOSE.

DECLARE

DECLARE names the cursor and associates it with a query. The query is not executed yet. 

EXEC SQL
    DECLARE EMP-CURSOR CURSOR FOR
        SELECT EMPID, EMPNAME, SALARY
          FROM EMPLOYEE
         WHERE DEPT = :WS-DEPT
         ORDER BY EMPID
END-EXEC.

The DECLARE is processed by the precompiler — it does not generate a runtime call. You can put it in WORKING-STORAGE alongside the host variable declarations.

OPEN / FETCH / CLOSE

PROCEDURE DIVISION.
    MOVE "ENG" TO WS-DEPT

    EXEC SQL OPEN EMP-CURSOR END-EXEC
    IF SQLCODE NOT = 0
        DISPLAY "OPEN failed: " SQLCODE
        STOP RUN
    END-IF

    PERFORM UNTIL SQLCODE = 100
        EXEC SQL
            FETCH EMP-CURSOR
             INTO :WS-EMP-ID, :WS-EMP-NAME, :WS-EMP-SALARY
        END-EXEC

        IF SQLCODE = 0
            DISPLAY WS-EMP-ID " " WS-EMP-NAME " " WS-EMP-SALARY
        END-IF
    END-PERFORM

    EXEC SQL CLOSE EMP-CURSOR END-EXEC

    STOP RUN.

Inside the loop, SQLCODE = 100 means "end of cursor" — the same role as AT END on a file read. After CLOSE, the cursor can be reopened with a fresh OPEN; this is how shops re-run the same query with different parameters.

Updatable cursors

A cursor declared FOR UPDATE can be used to update or delete the row currently positioned on: 

EXEC SQL
    DECLARE EMP-CURSOR CURSOR FOR
        SELECT EMPID, SALARY
          FROM EMPLOYEE
         WHERE DEPT = :WS-DEPT
         FOR UPDATE OF SALARY
END-EXEC.

* > inside the FETCH loop:
EXEC SQL
    UPDATE EMPLOYEE
       SET SALARY = SALARY * 1.05
     WHERE CURRENT OF EMP-CURSOR
END-EXEC.

This is more efficient than a separate UPDATE ... WHERE EMPID = :WS-EMP-ID because the database already has the row latched.

Error handling

Check SQLCODE after every call

This is the single most important habit. Every EXEC SQL sets SQLCODE, and ignoring it lets a quiet failure cascade into a much louder one several statements later.

The classification:

  • SQLCODE = 0 — success.
  • SQLCODE > 0 — warning. +100 means "no rows / end of cursor."
  • SQLCODE < 0 — error. The operation did not complete.

The codes you will hit most often

Code Meaning Typical cause
0 Success Operation completed normally.
+100 No rows / end of data SELECT INTO matched no rows; or FETCH walked off the end of a cursor.
-180 / -181 Invalid datetime Host variable does not contain a valid date/time string for its target column.
-530 Foreign key violation INSERT or UPDATE referenced a parent row that doesn't exist.
-803 Duplicate key INSERT or UPDATE would create a duplicate value in a unique index.
-805 Package not found Program was not bound to this database. Run BIND PACKAGE and try again.
-811 Row not unique SELECT INTO returned more than one row. Add a more specific WHERE, or convert to a cursor.
-911 Deadlock / timeout, rolled back Another transaction held the row. DB2 chose your transaction as the victim and rolled it back. Retry.
-913 Deadlock / timeout, not rolled back Same situation as -911 but only the current statement was rolled back. Retry.
-922 Authorization failure The runtime user does not have privilege for the operation.

The WHENEVER directive

Writing IF SQLCODE NOT = 0 after every call gets repetitive. The WHENEVER directive lets you set a default branch for each condition class: 

EXEC SQL WHENEVER SQLERROR GOTO ERR-HANDLER END-EXEC.
EXEC SQL WHENEVER NOT FOUND CONTINUE END-EXEC.
EXEC SQL WHENEVER SQLWARNING CONTINUE END-EXEC.

WHENEVER is a precompiler directive, not a runtime statement — it generates the equivalent IF SQLCODE ... GO TO after every subsequent EXEC SQL. Best practice is to use it sparingly; explicit checks are easier to trace.

Indicator variables and nulls

The problem

SQL has a notion of NULL — a tri-valued logic separate from the column's declared type. COBOL has no such concept; a numeric field is always some specific number, a character field is always some specific string of characters.

If a SQL column may be NULL and you fetch it directly into a COBOL host variable, the database has no way to tell you "this row's value is NULL." The result is SQLCODE = -305 (null value without indicator) and the host variable is untouched, which is a footgun — the COBOL variable now holds whatever it held before the FETCH.

The fix: an indicator variable

An indicator variable is a small COBOL field paired with each host variable. After the FETCH the database fills it in with:

  • 0 — the column value is in the host variable.
  • -1 — the column was NULL; the host variable was not touched.
  • -2 — the column was NULL because of a function failure (e.g. divide by zero).
  • positive — the value was truncated to fit (string columns).

Declared in the same DECLARE SECTION, named with the colon-comma-colon syntax in SQL:

EXEC SQL BEGIN DECLARE SECTION END-EXEC.
01  WS-EMP-COMMISSION    PIC S9(7)V99 COMP-3.
01  WS-COMM-IND          PIC S9(4)    COMP.
EXEC SQL END DECLARE SECTION END-EXEC.

EXEC SQL
    SELECT COMMISSION
      INTO :WS-EMP-COMMISSION :WS-COMM-IND
      FROM EMPLOYEE
     WHERE EMPID = :WS-EMP-ID
END-EXEC.

IF WS-COMM-IND = -1
    DISPLAY "Employee has no commission rate."
ELSE
    DISPLAY "Commission: " WS-EMP-COMMISSION
END-IF.

For INSERT in the other direction, set the indicator to -1 to write a NULL.

A complete example

The table

Run this once in your database to set the stage. The syntax is plain SQL, no COBOL involved. 

CREATE TABLE EMPLOYEE (
    EMPID      INTEGER       NOT NULL PRIMARY KEY,
    EMPNAME    VARCHAR(40)   NOT NULL,
    DEPT       CHAR(3)       NOT NULL,
    SALARY     DECIMAL(9,2)  NOT NULL,
    COMMISSION DECIMAL(9,2)
);

INSERT INTO EMPLOYEE VALUES (1001, 'Alice Smith',   'ENG', 75000.00, 5000.00);
INSERT INTO EMPLOYEE VALUES (1002, 'Bob Jones',     'ENG', 82000.00, NULL);
INSERT INTO EMPLOYEE VALUES (1003, 'Carol Khan',    'FIN', 95000.00, 8000.00);
INSERT INTO EMPLOYEE VALUES (1004, 'Dan Murphy',    'ENG', 65000.00, NULL);

COMMIT;

The program

🔍

Reads every employee in the ENG department, applies a 5% raise to those earning under 80,000, and writes a summary line for each. Demonstrates singleton SELECT, cursor iteration, positioned UPDATE, and an indicator variable on a nullable column. 

       IDENTIFICATION DIVISION.
       PROGRAM-ID. SALRAISE.

       DATA DIVISION.
       WORKING-STORAGE SECTION.

       EXEC SQL INCLUDE SQLCA END-EXEC.

       EXEC SQL BEGIN DECLARE SECTION END-EXEC.
       01  WS-EMP-ID          PIC S9(9)   COMP.
       01  WS-EMP-NAME        PIC X(40).
       01  WS-DEPT            PIC X(3).
       01  WS-SALARY          PIC S9(7)V99 COMP-3.
       01  WS-COMMISSION      PIC S9(7)V99 COMP-3.
       01  WS-COMM-IND        PIC S9(4)    COMP.
       EXEC SQL END DECLARE SECTION END-EXEC.

       EXEC SQL
           DECLARE EMP-CURSOR CURSOR FOR
               SELECT EMPID, EMPNAME, SALARY, COMMISSION
                 FROM EMPLOYEE
                WHERE DEPT = :WS-DEPT
                ORDER BY EMPID
                FOR UPDATE OF SALARY
       END-EXEC.

       PROCEDURE DIVISION.
       BEGIN-RUN.
           MOVE "ENG" TO WS-DEPT

           EXEC SQL OPEN EMP-CURSOR END-EXEC
           IF SQLCODE NOT = 0
               DISPLAY "OPEN failed, SQLCODE=" SQLCODE
               STOP RUN
           END-IF

           PERFORM UNTIL SQLCODE = 100
               EXEC SQL
                   FETCH EMP-CURSOR
                    INTO :WS-EMP-ID, :WS-EMP-NAME,
                         :WS-SALARY,
                         :WS-COMMISSION :WS-COMM-IND
               END-EXEC

               EVALUATE TRUE
                   WHEN SQLCODE = 100
                       CONTINUE
                   WHEN SQLCODE NOT = 0
                       DISPLAY "FETCH error " SQLCODE
                       EXIT PERFORM
                   WHEN WS-SALARY < 80000
                       EXEC SQL
                           UPDATE EMPLOYEE
                              SET SALARY = SALARY * 1.05
                            WHERE CURRENT OF EMP-CURSOR
                       END-EXEC
                       DISPLAY WS-EMP-NAME " raised."
                   WHEN OTHER
                       DISPLAY WS-EMP-NAME " held."
               END-EVALUATE
           END-PERFORM

           EXEC SQL CLOSE EMP-CURSOR END-EXEC
           EXEC SQL COMMIT END-EXEC
           STOP RUN.

The points to notice:

  • The cursor is declared in WORKING-STORAGE — that is the convention even though it reads like a procedural statement.
  • FOR UPDATE OF SALARY lets us use WHERE CURRENT OF EMP-CURSOR inside the loop — a positioned update.
  • The indicator variable on COMMISSION means we can fetch the column without crashing on NULL rows.
  • The explicit COMMIT at the end persists the updates. Without it the database may roll them back at program termination depending on the runtime.

DB2 specifics

The BIND step

The DB2 precompiler emits two things: a processed COBOL source file (input to the compiler) and a DBRM (DataBase Request Module) containing the SQL text. After the program is compiled and linked, you BIND the DBRM into the database.

BIND is where DB2 picks access paths: which index to use for each query, in what order to join tables, when to materialise results. The optimiser does this work once, at bind time, not every time the program runs. The output is a package stored in the database. 

BIND PACKAGE(MYPKG) MEMBER(SALRAISE) ACTION(REPLACE) -
     ISOLATION(CS) RELEASE(COMMIT) VALIDATE(BIND)

When the program runs, it loads the package by name and uses the pre-built access paths.

Plans and packages

Two concepts you will see in DB2 documentation:

  • Package — the bound form of one program's SQL. One per program.
  • Plan — a named collection of packages that a runtime connection uses. Modern shops use trivial plans that contain a single package list.

The runtime invocation specifies a plan; the plan resolves package names to actual SQL.

What about Oracle?

Oracle's equivalent is Pro*COBOL. The EXEC SQL ... END-EXEC, DECLARE SECTION, cursor lifecycle, and host-variable syntax are essentially identical — this is what the ANSI standard for embedded SQL guarantees. The differences are:

  • No BIND step. Oracle parses the SQL at runtime (with a statement cache) instead of binding it ahead of time.
  • Slightly different error codes. Oracle has its own ORA-xxxxx series. The shape of the checks (SQLCODE < 0) is the same.
  • SQLCA is essentially the same. Use it the same way.

If you can read DB2 embedded SQL you can read Pro*COBOL after about an hour of looking up the deltas.

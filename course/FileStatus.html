Introduction

What this page is

The file-handling chapters (sequential, relative, indexed) introduce FILE STATUS and discuss the two codes you will see most often: 00 (success) and 10 (end of file). This page is the lookup table for everything else.

It is reference material, not a tutorial. Bookmark it for the next time your program returns 35.

Declaring FILE STATUS

You opt in to file-status reporting by adding a FILE STATUS IS clause to the SELECT: 

SELECT StudentFile  ASSIGN TO "STUDENTS.DAT"
                    ORGANIZATION IS SEQUENTIAL
                    FILE STATUS IS WS-Stud-Status.

WORKING-STORAGE SECTION.
01  WS-Stud-Status      PIC X(2).

After every I/O verb (OPEN, READ, WRITE, REWRITE, DELETE, CLOSE, START), the runtime stores a two-character status into WS-Stud-Status. The first byte is the major category; the second byte is a sub-status that refines the meaning.

How to read the tables

Each major category section below contains a table with three columns: the two-character code, the standard meaning, and the typical cause. Codes are listed in numerical order within each category. Where compilers commonly differ I have noted it.

Category 0x — successful completion

The good news

🔍

The first byte 0 means the operation succeeded. A non-zero second byte is a warning rather than a failure — the operation completed, but there is something worth knowing.

Code Meaning Typical cause
00 Successful completion No warnings — the verb completed normally.
02 Duplicate alternate key On an indexed file: the verb succeeded, but the record matches another record on a non-primary key declared WITH DUPLICATES.
04 Record length mismatch On READ: the verb succeeded, but the record's actual length doesn't match the FD entry's. Often indicates a corrupted file or a wrong record layout.
05 OPEN: file did not exist On OPEN OUTPUT / OPEN I-O / OPEN EXTEND: the file was created. Not an error; informational.
07 OPEN: device incompatible The file was opened, but the ASSIGN target is a non-reel-mountable device (typical for a printer assigned as a tape). Rarely seen on modern systems.

Category 1x — at end

End-of-data conditions

Code Meaning Typical cause
10 End of file On READ: there are no more records. The equivalent of the AT END clause being triggered.
14 Relative key too large On a relative-file READ NEXT: the relative key would exceed the declared size. Effectively also an "at end" but specific to relative.

Category 2x — invalid key

Indexed and relative file key problems

Code Meaning Typical cause
21 Sequence error On a sequentially-accessed indexed file: a WRITE or REWRITE would produce records out of key order.
22 Duplicate key on a unique key WRITE on an indexed file would create a duplicate value on the primary key (or on an alternate key not declared WITH DUPLICATES).
23 Record not found On READ, START, REWRITE, DELETE: no record matches the key. By far the most common 2x code in practice.
24 Boundary violation WRITE on a relative file: the relative key exceeds the configured maximum. Or, on an indexed file: the file's allocated space has been exhausted.

Category 3x — permanent error

I/O system errors

Code Meaning Typical cause
30 Permanent error, unspecified A catch-all when the runtime cannot map the underlying I/O error to a more specific code. Look in the system log.
34 Boundary violation, sequential On a sequential file: the WRITE would exceed the file's allocated space. The mainframe equivalent is an S001 abend.
35 File not found On OPEN INPUT / OPEN I-O: the file does not exist. Almost always: wrong path, wrong ddname, or a missing JCL allocation.
37 Permission / device error OPEN failed because of an underlying device or permission error. Check the system log and dataset RACF profile if on z/OS.
38 OPEN of a closed-with-lock file A previous step closed the file with CLOSE WITH LOCK; this step is not allowed to reopen it.
39 File attribute mismatch On OPEN: the file's actual record format, record length, or block size does not match the COBOL FD declaration. On z/OS this is the most common cause of an S013 abend.

Category 4x — logic error

Program-level mistakes

Code Meaning Typical cause
41 OPEN on an already-open file The program issued OPEN twice without an intervening CLOSE.
42 CLOSE on an unopened file Often the result of a control-flow bug where the OPEN path didn't run.
43 REWRITE / DELETE without a preceding READ On a sequentially-accessed file you must READ a record before you can REWRITE or DELETE it.
44 REWRITE record length mismatch On a variable-length file: REWRITE with a record whose length differs from the original. Some compilers allow this; the standard does not.
46 READ on a file at end Another READ was issued after 10 had already been returned, without a re-positioning START.
47 READ on a file not opened for input The file was opened OPEN OUTPUT or OPEN EXTEND; you cannot READ from it.
48 WRITE on a file not opened for output Mirror of 47: the file was opened OPEN INPUT; you cannot WRITE to it.
49 DELETE / REWRITE on a file not opened I-O Both verbs require OPEN I-O.

Category 9x — implementor-defined

Vendor-specific

The 9x codes are explicitly carved out by the COBOL standard for compiler vendors to use. Values and meanings differ between IBM Enterprise COBOL, OpenText (Micro Focus) Visual COBOL, GnuCOBOL, and Fujitsu. The common patterns:

Code IBM Enterprise COBOL GnuCOBOL
90 Unspecified runtime error (reserved)
91 VSAM password / RACF failure File not available (locked by another process)
92 Logic error (variant of 41-49) Logic error
93 Resource not available (varies)
94 VSAM logic error (varies)
95 File-info inconsistency (varies)
96 No DD statement (N/A)
97 OPEN successful, file integrity verified (varies)
98 File-locking failure (varies)
99 Record-locking failure (varies)

If you are working against a specific compiler, its manual will have the authoritative list. For IBM the reference is the Enterprise COBOL for z/OS Language Reference; for GnuCOBOL it is the runtime documentation under cob_fcd_status_codes.

Common scenarios

"What does this code mean for me?"

Reverse-lookup of the codes you will hit most often, by the verb that produced them:

Verb Code Most likely cause
OPEN INPUT 35 File doesn't exist at the configured path / ddname.
OPEN INPUT 37 Permission denied (on z/OS: RACF dataset profile).
OPEN INPUT 39 RECFM/LRECL/BLKSIZE on the file don't match the COBOL FD.
READ (sequential) 10 End of file. Trigger the AT END branch.
READ (indexed, random) 23 No record with that key.
WRITE (indexed) 22 Duplicate key on a unique-key index.
WRITE (sequential) 34 Output dataset ran out of space.
REWRITE / DELETE 43 Sequential access mode — no preceding READ.
REWRITE / DELETE 49 File was opened INPUT or OUTPUT, not I-O.
START 23 No record matches the search condition.

Best practice

The canonical pattern

The discipline is: declare FILE STATUS on every SELECT, check it after every I/O verb, and abend with a clear message if anything other than the expected value comes back. 

       ENVIRONMENT DIVISION.
       INPUT-OUTPUT SECTION.
       FILE-CONTROL.
           SELECT EmpFile  ASSIGN TO EMPLOYEE
                           ORGANIZATION IS INDEXED
                           ACCESS MODE  IS DYNAMIC
                           RECORD KEY   IS EmpId
                           FILE STATUS  IS WS-FS.

       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  WS-FS                PIC X(2).
           88  FS-OK            VALUE "00".
           88  FS-EOF           VALUE "10".
           88  FS-DUP-KEY       VALUE "22".
           88  FS-NOT-FOUND     VALUE "23".
           88  FS-FILE-MISSING  VALUE "35".

       PROCEDURE DIVISION.
       BEGIN-RUN.
           OPEN INPUT EmpFile
           EVALUATE TRUE
               WHEN FS-OK
                   CONTINUE
               WHEN FS-FILE-MISSING
                   DISPLAY "EMPLOYEE file not found - check JCL"
                   MOVE 8 TO RETURN-CODE
                   GOBACK
               WHEN OTHER
                   DISPLAY "OPEN failed with status " WS-FS
                   MOVE 12 TO RETURN-CODE
                   GOBACK
           END-EVALUATE.

           READ EmpFile
           PERFORM UNTIL FS-EOF
               IF FS-OK
                   PERFORM Process-Record
               ELSE
                   DISPLAY "READ failed with status " WS-FS
                   MOVE 12 TO RETURN-CODE
                   GOBACK
               END-IF
               READ EmpFile
           END-PERFORM

           CLOSE EmpFile
           STOP RUN.

The two patterns worth absorbing:

  • 88-level condition names on the status field. Reading IF FS-OK is much easier than IF WS-FS = "00", and the names self-document.
  • Set the RETURN-CODE before GOBACK when you abort on an unexpected status. On z/OS this is what COND= in subsequent JCL steps will branch on. See An introduction to JCL for the JCL side.

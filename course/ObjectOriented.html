Introduction

Aims

The COBOL 2002 standard added an object-oriented dialect on top of the procedural core. New keywords (CLASS-ID, METHOD-ID, INVOKE, INHERITS FROM), new data types (USAGE OBJECT REFERENCE), and a new program structure (a class is not a program; methods are not paragraphs).

This unit teaches you to read OO COBOL and explains where you will actually meet it. It is optional. Most production code — including everything earlier in this course — is procedural and will stay procedural for the foreseeable future.

Objectives

By the end of this unit you should —

  1. Be able to read an OO COBOL class definition and identify its methods and instance data.

  2. Understand the difference between CALL and INVOKE.

  3. Recognise inheritance and overriding when you see them.

  4. Know when OO COBOL is the right tool — primarily JVM and .NET interop — and when it is the wrong one.

Prerequisites

This is an end-of-course chapter; comfort with the procedural language is assumed.

  • The structure of COBOL programs

History and scope

How OO got into COBOL

The 1985 standard had no OO at all. Through the late 1990s individual vendors (IBM, Micro Focus, Fujitsu) added their own OO extensions, each subtly different. ISO/IEC 1989:2002 standardised a common dialect, drawing heavily on the Micro Focus design. ISO/IEC 1989:2014 tightened and extended it. 2023 added a small number of features.

Compiler support today:

Compiler OO support
IBM Enterprise COBOL for z/OS Yes, since v3. Tight integration with JNI for Java interop.
OpenText (Micro Focus) Visual COBOL Yes. Strong integration with .NET and JVM — classes can implement Java or .NET interfaces directly.
Fujitsu NetCOBOL Yes, with its own .NET integration.
GnuCOBOL Partial. CLASS-ID / METHOD-ID exist but the runtime is limited. Treat as experimental.

CLASS-ID and METHOD-ID

The shape of a class

A class definition replaces a program. Where a procedural source file starts with IDENTIFICATION DIVISION. PROGRAM-ID., a class source file starts with IDENTIFICATION DIVISION. CLASS-ID. Methods replace paragraphs. 

       IDENTIFICATION DIVISION.
       CLASS-ID. Account.

       *> Instance data: per-object state.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  Balance              PIC S9(9)V99 COMP-3 VALUE ZERO.

       *> Methods follow.

       IDENTIFICATION DIVISION.
       METHOD-ID. Deposit.

       DATA DIVISION.
       LINKAGE SECTION.
       01  LK-Amount            PIC S9(9)V99 COMP-3.

       PROCEDURE DIVISION USING BY VALUE LK-Amount.
           ADD LK-Amount TO Balance.

       END METHOD Deposit.

       IDENTIFICATION DIVISION.
       METHOD-ID. Withdraw.

       DATA DIVISION.
       LINKAGE SECTION.
       01  LK-Amount            PIC S9(9)V99 COMP-3.
       01  LK-Success           PIC X.

       PROCEDURE DIVISION USING BY VALUE LK-Amount
                          RETURNING LK-Success.
           IF Balance >= LK-Amount
               SUBTRACT LK-Amount FROM Balance
               MOVE "Y" TO LK-Success
           ELSE
               MOVE "N" TO LK-Success
           END-IF.

       END METHOD Withdraw.

       IDENTIFICATION DIVISION.
       METHOD-ID. GetBalance.

       DATA DIVISION.
       LINKAGE SECTION.
       01  LK-Balance           PIC S9(9)V99 COMP-3.

       PROCEDURE DIVISION RETURNING LK-Balance.
           MOVE Balance TO LK-Balance.

       END METHOD GetBalance.

       END CLASS Account.

Notice the shape:

  • Instance data is in WORKING-STORAGE. Each instance gets its own copy.
  • Method parameters use PROCEDURE DIVISION USING ... RETURNING ... — the same syntax as a subprogram, declared in LINKAGE SECTION.
  • The terminator is END METHOD for methods and END CLASS at the very end.

INVOKE

Calling a method

INVOKE is to objects what CALL is to subprograms. It takes an object reference, a method name, and parameters. 

       WORKING-STORAGE SECTION.
       01  MyAccount        USAGE OBJECT REFERENCE Account.
       01  WS-OK            PIC X.
       01  WS-Balance       PIC S9(9)V99 COMP-3.

       PROCEDURE DIVISION.
           *> Create a new instance and store it.
           INVOKE Account "NEW" RETURNING MyAccount

           *> Call methods on it.
           INVOKE MyAccount "Deposit" USING BY VALUE 100.00
           INVOKE MyAccount "Withdraw"
                  USING BY VALUE 40.00
                  RETURNING WS-OK
           INVOKE MyAccount "GetBalance" RETURNING WS-Balance
           DISPLAY "Balance is " WS-Balance.

The string literal "NEW" in the first INVOKE calls the class-side factory method that constructs an instance. It's the equivalent of new Account() in Java.

The compiler resolves the method name at compile time when the object reference is typed (OBJECT REFERENCE Account) or dispatches dynamically at run time when the reference is generic (OBJECT REFERENCE with no class name — the Object root type).

Inheritance

INHERITS FROM

A subclass declares its parent on the CLASS-ID line: 

       IDENTIFICATION DIVISION.
       CLASS-ID. SavingsAccount INHERITS FROM Account.

       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  InterestRate         PIC S9(3)V9(6) COMP-3 VALUE 0.025.

       IDENTIFICATION DIVISION.
       METHOD-ID. AccrueInterest.

       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  WS-CurrentBalance    PIC S9(9)V99 COMP-3.
       01  WS-Interest          PIC S9(9)V99 COMP-3.

       PROCEDURE DIVISION.
           INVOKE SELF "GetBalance" RETURNING WS-CurrentBalance
           COMPUTE WS-Interest = WS-CurrentBalance * InterestRate
           INVOKE SELF "Deposit" USING BY VALUE WS-Interest.

       END METHOD AccrueInterest.

       END CLASS SavingsAccount.

SELF is the equivalent of this in Java or C# — an object reference to the currently executing instance.

To override an inherited method, define a method with the same name in the subclass. Method dispatch is dynamic by default — the runtime picks the most-derived implementation for the actual object's class.

OBJECT REFERENCE data items

Typed and untyped references

A data item declared USAGE OBJECT REFERENCE holds either NULL or a reference to an object. There are two flavours:

  • Typed — USAGE OBJECT REFERENCE Account. The compiler knows which methods are valid; calls to nonexistent methods fail at compile time.
  • Untyped — USAGE OBJECT REFERENCE. The reference can point at any class; method dispatch is fully dynamic.

Object references obey the same lifetime rules as any other data item: they live in WORKING-STORAGE for the duration of the program (or method) they're declared in. The objects they point at are garbage-collected by the OO COBOL runtime.

Factory methods

Class-side behaviour

A factory method belongs to the class itself, not to instances of it — the OO COBOL equivalent of a static method. They're useful for constructing instances with non-trivial setup, or for sharing class-level state. 

       IDENTIFICATION DIVISION.
       CLASS-ID. Account.

       FACTORY.

       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  NextAccountNumber    PIC S9(9) COMP VALUE 1.

       IDENTIFICATION DIVISION.
       METHOD-ID. CreateNumbered.

       DATA DIVISION.
       LINKAGE SECTION.
       01  LK-NewAccount        USAGE OBJECT REFERENCE Account.

       PROCEDURE DIVISION RETURNING LK-NewAccount.
           INVOKE Account "NEW" RETURNING LK-NewAccount
           *> Set the new account's ID, increment the counter, etc.
           ADD 1 TO NextAccountNumber.

       END METHOD CreateNumbered.

       END FACTORY.

       *> Instance section follows...
       OBJECT.

       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  Balance              PIC S9(9)V99 COMP-3 VALUE ZERO.

       *> Instance methods...

       END OBJECT.

       END CLASS Account.

The class definition is bracketed by FACTORY/END FACTORY for class-side state and methods, and OBJECT/END OBJECT for instance-side. Code calls INVOKE Account "CreateNumbered" — using the class name itself as the receiver.

Java and .NET interop

The actual use case

The most common reason a shop touches OO COBOL is interoperating with code in another language — usually Java on z/OS or .NET on Windows.

  • IBM Enterprise COBOL on z/OS can call Java classes and be called from Java. The mechanism is JNI under the hood; OO COBOL syntax is what shields the application programmer from it.
  • OpenText Visual COBOL compiles to either JVM bytecode or .NET IL. Classes written in OO COBOL can directly implement Java or .NET interfaces; instances are usable from any other language on the same runtime.

In both cases the OO COBOL is essentially a bridge: a thin object layer in front of the business logic so that another language can hold and call it.

When to use it

The honest guidance

OO COBOL is the right tool in a small number of specific situations:

  • You need to expose existing COBOL business logic to a Java or .NET caller and don't want to bridge through JNI / P/Invoke by hand.
  • You're writing new code on Visual COBOL for the JVM or .NET and the surrounding ecosystem is OO.
  • A specific framework (e.g. some IBM modernisation tools) requires class-shaped entry points.

For everything else — including everything else covered in this course — procedural COBOL is the right answer. The procedural language is more thoroughly tested across compilers, more familiar to every other engineer who will touch the code, and equally able to express any business algorithm.

If your team is debating whether to write a new module in OO COBOL, the default answer should be "no, write it procedurally" unless one of the bullets above clearly applies.

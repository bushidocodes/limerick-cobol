Introduction

Aims

The rest of this course teaches the batch style of COBOL — programs that open files, process records, and exit. That is half of the COBOL world. The other half is online transaction processing: short-lived programs that handle a single screen interaction at a time, run thousands of concurrent users, and are coordinated by a transaction monitor called CICS (Customer Information Control System).

This unit gives you the mental model. It will not make you a CICS programmer — that takes longer than one chapter — but after reading it you should be able to read CICS source code and understand the shape of the program.

Objectives

By the end of this unit you should —

  1. Understand what a transaction monitor is and what role CICS plays on z/OS.

  2. Be able to explain pseudo-conversational programming and contrast it with batch and conversational designs.

  3. Recognise the core EXEC CICS commands and what they correspond to in your existing COBOL vocabulary.

  4. Understand the role of BMS maps and how they connect to COBOL data structures.

  5. Know to check EIBRESP after every CICS call, and recognise common return values.

Prerequisites

  • The structure of COBOL programs

  • Contained and external sub-programs
  • Optional but useful: An introduction to JCL for the surrounding mainframe context.

What CICS is

The transaction monitor

CICS is a long-running server program (or set of programs called regions) that sits between the network and your COBOL code. It provides services that batch COBOL doesn't need:

  • Multitasking. A single CICS region runs many concurrent transactions. Each transaction is a short-lived invocation of a COBOL program.
  • Screen I/O. CICS owns the connection to the terminal (or, today, the web client). Your COBOL program does not call sockets or 3270 protocol; it asks CICS to send and receive screens.
  • Resource locking and recovery. CICS coordinates access to shared files and databases. If a transaction abends, CICS rolls back the changes it made.
  • Security. CICS authenticates users and authorises each transaction against site security profiles.
  • Performance. CICS keeps program load modules in memory, pools threads, and generally avoids the per-request startup cost a batch design would pay.

In modern shops the front-end may be a web browser, a mobile app, or an API client — but the COBOL program on the back end still looks like a CICS transaction, just with the screen replaced by JSON or XML.

A CICS transaction

At the user's end a transaction is one unit of work — press Enter, see a response. At the system's end it is one invocation of a CICS-aware program, identified by a four-character transaction ID.

The CICS administrator configures, in the system tables, that transaction ID EMPS runs program EMPSRCH. When a user types EMPS on a 3270 screen, CICS:

  1. Looks up the transaction ID in its tables.
  2. Loads program EMPSRCH if it isn't already in memory.
  3. Sets up the execution environment (EIB, COMMAREA pointer, …).
  4. Branches into the program's entry point.

When the program runs EXEC CICS RETURN, the transaction ends and CICS goes back to waiting for input. The program is unloaded (or just released back to the pool); the next invocation starts from scratch.

Pseudo-conversational programming

Three styles

There are three ways a CICS program can interact with the user:

  • Non-conversational. The program never waits for the user. It runs once, produces output, and returns. Like a one-shot batch job, just dispatched by CICS instead of JCL.
  • Conversational. The program sends a screen, calls EXEC CICS RECEIVE to block until the user replies, and continues running. While waiting, it holds memory and locks. Don't write these. They scale badly and are almost always wrong.
  • Pseudo-conversational. The program sends a screen and immediately returns — it does not wait. When the user presses Enter, CICS starts a fresh invocation of the program (or a different program), which reads the response and continues from the logical next step.

Pseudo-conversational is the one you write. Almost all production CICS COBOL uses it.

The shape of a pseudo-conversational program

🔍

This is the mental shift. Each invocation of the program is independent. To remember anything across invocations you must explicitly save it in storage that CICS preserves — the COMMAREA (Part 3) or a temporary storage queue.

A typical program looks like this:

       PROCEDURE DIVISION.
       BEGIN-RUN.
           IF EIBCALEN = 0
               PERFORM First-Time-Through
           ELSE
               PERFORM Subsequent-Pass
           END-IF
           PERFORM Return-To-Cics.

       First-Time-Through.
           * > Show the initial empty form.
           EXEC CICS SEND MAP("EMPMAP") MAPSET("EMPSET") ERASE END-EXEC
           MOVE "DISPLAY" TO WS-Next-Step.

       Subsequent-Pass.
           EXEC CICS RECEIVE MAP("EMPMAP") MAPSET("EMPSET") END-EXEC
           EVALUATE WS-Next-Step
               WHEN "DISPLAY"  PERFORM Look-Up-Employee
               WHEN "EDIT"     PERFORM Save-Changes
               WHEN OTHER      PERFORM Show-Error
           END-EVALUATE.

       Return-To-Cics.
           EXEC CICS RETURN
               TRANSID("EMPS")
               COMMAREA(WS-Commarea)
               LENGTH(LENGTH OF WS-Commarea)
           END-EXEC.

EIBCALEN is "EIB COMMAREA length" — it is zero on the very first invocation, non-zero on every subsequent one because the previous invocation passed a COMMAREA when it returned.

The TRANSID parameter on EXEC CICS RETURN tells CICS "next time this user presses Enter, dispatch transaction EMPS — that is, run me again." Without it the user's terminal goes back to the application menu.

The COMMAREA

The forget-me-not

A COMMAREA (communication area) is a block of memory the program declares in its LINKAGE SECTION. When the program issues EXEC CICS RETURN with a COMMAREA parameter, CICS saves the block. When the next invocation of the same transaction ID starts, CICS hands it back to the new instance. 

       LINKAGE SECTION.
       01  DFHCOMMAREA.
           05  WS-Next-Step          PIC X(10).
           05  WS-Selected-Emp       PIC 9(7).
           05  WS-Filter-Text        PIC X(40).

       PROCEDURE DIVISION.
           IF EIBCALEN = 0
               INITIALIZE DFHCOMMAREA
           END-IF
           ...
           EXEC CICS RETURN
               TRANSID("EMPS")
               COMMAREA(DFHCOMMAREA)
               LENGTH(LENGTH OF DFHCOMMAREA)
           END-EXEC.

The name DFHCOMMAREA is conventional — CICS sets up the addressability for you. You declare the structure under it in the LINKAGE SECTION.

What goes in the COMMAREA

Practical rules of thumb:

  • State the next invocation needs to do its job. Which step we are on, which row the user selected, any field they typed that hasn't been saved yet.
  • Not large blobs. COMMAREAs are bounded (~32 KB in older CICS, larger in newer versions). Large state goes into a temporary storage queue, not the COMMAREA.
  • Not anything secret you couldn't store on the wire. COMMAREA passes through CICS memory; on traditional 3270 setups it doesn't leak, but in modern Web Services scenarios it can. Treat it as transport.

EXEC CICS commands

The core set

CICS exposes its services to your program through EXEC CICS ... END-EXEC blocks, processed by a preprocessor analogous to the SQL precompiler (see Embedded SQL in COBOL). The verbs cluster by responsibility:

Category Verb What it does
Screen SEND MAP Push a formatted screen out to the terminal.
RECEIVE MAP Read user input from a formatted screen into COBOL data items.
File / VSAM READ FILE Random read by key.
WRITE FILE Insert a record.
REWRITE Update a record previously READ FOR UPDATE.
DELETE Delete by key.
Program flow LINK Call another program, then return here. Like a COBOL CALL.
XCTL Transfer control to another program; do not return. Like a chained GOTO across programs.
RETURN End this invocation. Pass a COMMAREA back if the transaction continues.
Recovery SYNCPOINT Commit changes made so far.
SYNCPOINT ROLLBACK Roll back changes made so far.
Storage GETMAIN Allocate working storage at runtime.
FREEMAIN Release storage allocated with GETMAIN.

The FILE verbs are functionally similar to native COBOL file I/O verbs (indexed files) but go through CICS so the I/O participates in transaction recovery.

BMS maps

Where the screen lives

A CICS COBOL program does not contain the text of the screens it sends. The screen layout lives in a separate BMS (Basic Mapping Support) map source file, processed by an assembler macro pass that produces two outputs:

  • A physical map — a load module the CICS region uses at runtime to send and receive the screen.
  • A symbolic map — a COBOL COPY book of data items that mirror each field on the screen.

The program COPYs the symbolic map and then references its fields directly:

       WORKING-STORAGE SECTION.
       01  WS-Emp-Map.
           COPY EMPSET.            *> symbolic map

       PROCEDURE DIVISION.
           EXEC CICS RECEIVE MAP("EMPMAP") MAPSET("EMPSET")
                              INTO(WS-Emp-Map)
           END-EXEC.

           IF EMPIDI > "0000000"
              MOVE EMPIDI TO WS-Search-Key
              ...
           END-IF.

Field names follow a convention: a field called EMPID on the map produces EMPIDI (input value), EMPIDO (output value), EMPIDA (attribute byte), and EMPIDL (entered length) in the symbolic map. The O suffix is what you set before SEND MAP; the I suffix is what comes back from RECEIVE MAP.

The BMS source itself is assembler macros, not COBOL, and a full discussion is beyond this chapter. The important point: the screen layout is a separate artifact that the program references by name.

EIBRESP and error handling

The EIB

CICS gives every program a free EIB (Exec Interface Block) — a record of information about the current transaction. The most useful fields:

Field Meaning
EIBCALEN Length of the COMMAREA passed in (0 on first invocation).
EIBRESP Return code from the most recent EXEC CICS command.
EIBRESP2 Sub-code refining EIBRESP.
EIBTRNID The transaction ID currently running.
EIBTASKN The task number CICS assigned. Useful for logs.
EIBTRMID Terminal ID the user is on.

Checking EIBRESP

The discipline mirrors what you already know from SQLCODE and FILE STATUS: check EIBRESP after every EXEC CICS. 

EXEC CICS READ
    FILE("EMPMAST")
    INTO(WS-Emp-Rec)
    RIDFLD(WS-Search-Key)
    RESP(WS-Resp) RESP2(WS-Resp2)
END-EXEC.

EVALUATE WS-Resp
    WHEN DFHRESP(NORMAL)
        PERFORM Display-Employee
    WHEN DFHRESP(NOTFND)
        PERFORM Show-Not-Found-Message
    WHEN OTHER
        PERFORM Critical-Error-Handler
END-EVALUATE.

The DFHRESP(NORMAL) macro expands to the integer constant for NORMAL. Using the macros instead of bare numbers is the convention — it makes the code legible and survives any future renumbering.

If you don't pass RESP on the verb, CICS will instead raise a condition handled by the most recent EXEC CICS HANDLE CONDITION or trigger the default abend. Most modern code uses the explicit RESP style.

Common RESP codes

Symbol Meaning
NORMAL Success.
NOTFND READ: no record matches the key.
DUPREC WRITE: duplicate key.
MAPFAIL RECEIVE MAP: the user pressed Enter on an empty screen, or the map name is wrong.
PGMIDERR LINK / XCTL: target program not defined in CICS tables.
FILENOTFOUND READ: the file is not defined in CICS tables.
NOSPACE WRITE: file is full.
NOTAUTH The transaction user is not authorised for this operation.
QIDERR Temporary-storage queue not found.

What this chapter does not cover

Honest scope

CICS is a deep subject. The topics deliberately left out of this primer, and where to read them:

  • Full BMS map authoring. The assembler-macro syntax, field attributes, MDT handling, and physical-map versus symbolic-map mechanics. See IBM's CICS Application Programming Guide.
  • Resource Definition Online (RDO). How transactions, programs, files, and queues are actually defined in the CICS region tables. Usually a CICS sysprog's job, not the application programmer's.
  • CICS Web Services. Exposing a CICS COBOL program as a SOAP or REST service via z/OS Connect. Increasingly the way modern CICS applications are accessed.
  • Channels and containers. The modern alternative to the COMMAREA for passing larger and more structured data between linked programs.
  • Transaction recovery details. Sync points, recoverable resources, in-doubt transactions, two-phase commit with DB2.

Where to go next

Once the mental model is in place, the most useful follow-up reads are IBM's free CICS Application Programming Guide and Doug Lowe's Murach's CICS Desk Reference.

If you came here from the COBOL chapters and want to ground the transaction model in batch terms, the most analogous pair is Embedded SQL in COBOL — the same EXEC X ... END-EXEC shape, the same return-code discipline, the same role for a precompiler.

Introduction

Aims

The remainder of this course teaches the COBOL language. Before you can usefully read any of it, you need to be able to compile and run a COBOL program on your own machine.

This unit walks you through three choices — which compiler, which editor, and which platform — and then has you compile and run a short complete program end-to-end.

If you already have a working COBOL toolchain you can skim this page and jump to The structure of COBOL programs.

Objectives

By the end of this unit you should —

  1. Be able to choose a COBOL compiler appropriate to your platform and goals.

  2. Have GnuCOBOL (or an equivalent) installed and verified on your machine.

  3. Be able to compile and run a short COBOL program from a terminal.

  4. Recognise the most common first-run errors and know how to fix them.

Prerequisites

  • None — this is the entry point of the course.
  • Comfort using a terminal (PowerShell on Windows; Terminal on macOS; any shell on Linux) is helpful but not required.

Choosing a compiler

Why this choice matters

COBOL is a standardised language — the latest published standard is ISO/IEC 1989:2023 — but no two compilers implement exactly the same subset. The compiler you pick determines:

  • Which platforms you can target (Windows, macOS, Linux, z/OS, AIX, …).
  • Which language features are available (free-format source, intrinsic functions, JSON PARSE, object-oriented COBOL, …).
  • How you compile and link (`cobc`, IBM JCL, Visual Studio, …).
  • The cost — some compilers are free; others are commercial.

For the purposes of this course any standards-compliant compiler will work. The example programs are written against COBOL 85 with a small number of COBOL 2002 extensions (free-format source, intrinsic functions), and they compile cleanly on every compiler listed below.

The shortlist

Compiler Platforms Cost Best for
GnuCOBOL (formerly OpenCOBOL) Windows, macOS, Linux, BSD Free, open-source (GPL) Learning, hobby projects, anything off-mainframe. Recommended for this course.
OpenText (Micro Focus) Visual COBOL Windows, Linux, AIX, .NET, JVM Commercial; free trial available Enterprise development outside z/OS — banks and insurance shops that have moved off-mainframe.
IBM Enterprise COBOL for z/OS z/OS only Commercial; bundled with z/OS Anything you intend to run on a real mainframe in production. The dominant compiler in banking, insurance, and government.
TutorialsPoint online compiler Browser Free Quickly trying a snippet without installing anything. Not appropriate for multi-file programs or file I/O.

If you are not sure which to use, install GnuCOBOL. It is free, it runs on every desktop platform, and the rest of this page assumes it.

Installing GnuCOBOL

Windows

The most reliable way to get GnuCOBOL on Windows is via MSYS2, which provides a Unix-like environment and a package manager.

  1. Install MSYS2 from https://www.msys2.org/. Accept the default install path.
  2. Open the MSYS2 UCRT64 shell from the Start menu.
  3. Update the package database: 
    pacman -Syu
    Close and re-open the shell if prompted.
  4. Install GnuCOBOL: 
    pacman -S mingw-w64-ucrt-x86_64-gnucobol
  5. Verify: 
    cobc --version
    You should see something like cobc (GnuCOBOL) 3.2.

Alternative on Windows: the Chocolatey package manager has a gnucobol package (choco install gnucobol) but it tends to lag behind the MSYS2 build.

macOS

The Homebrew formula is the simplest path.

  1. Install Homebrew if you don't have it — instructions at https://brew.sh/.
  2. Install GnuCOBOL: 
    brew install gnu-cobol
  3. Verify: 
    cobc --version

Linux

Most distributions ship GnuCOBOL in their default repositories.

Debian / Ubuntu / Mint:

sudo apt update
sudo apt install gnucobol

Fedora / RHEL / Rocky:

sudo dnf install gnucobol

Arch / Manjaro: available via the AUR as gnucobol.

Verify the install the same way:

cobc --version

What you just installed

The GnuCOBOL package gives you three things:

  • cobc — the COBOL compiler driver. This is the command you will run to turn .cob files into executables.
  • cobcrun — a small program that loads and runs a compiled COBOL module. You will use this less often than cobc.
  • A runtime library (libcob) that your compiled COBOL programs link against.

You do not need to install anything else specific to COBOL. The compiler emits C code under the hood and uses your system C compiler (GCC on Linux, Clang on macOS, MinGW-w64 on Windows) to produce the final executable — all of this is invoked automatically by cobc.

Choosing an editor

Recommended: VS Code + COBOL extension

Visual Studio Code with the COBOL extension by bitlang is the simplest modern editor setup. To install:

  1. Install VS Code from code.visualstudio.com.
  2. Open VS Code, press Ctrl+Shift+X (⌘+Shift+X on macOS) to open the Extensions panel.
  3. Search for COBOL and install the extension by bitlang.

The extension provides:

  • Syntax highlighting for .cob, .cbl, and .cpy files.
  • Outline view showing divisions, sections, and paragraphs.
  • Snippets for common patterns (PERFORM UNTIL, file SELECT clauses, …).
  • Folding by COBOL block.

For source-level debugging on Windows or Linux, also install the companion GnuCOBOL Debugger extension; on macOS use gdb manually as the debugger story for COBOL on Apple Silicon is still maturing.

Alternative editors

Anything with adjustable tab stops and a monospaced font will work. The lessons in this course use fixed-format COBOL, where columns 1–6, 7, 8–11, and 12–72 each have a defined meaning — so a text editor that can show a column ruler helps.

  • JetBrains IntelliJ IDEA / Rider — the bitlang-cobol plugin gives a similar experience to VS Code.
  • Vim / Neovim — built-in cobol filetype with syntax highlighting.
  • Emacs — cobol-mode from the Emacs package archives.
  • Notepad++ — ships with a COBOL lexer.

Hello, World

The program

🔍

The leading blank columns in this example are significant. COBOL in fixed format (the default) reserves columns 1–6 for an optional sequence number and column 7 for a status indicator (* for a comment, - for a continuation line, blank for a normal line). Code starts at column 8.

Open your editor and create a file named hello.cob with the following contents: 

       IDENTIFICATION DIVISION.
       PROGRAM-ID. Hello.

       PROCEDURE DIVISION.
           DISPLAY "Hello, World!".
           STOP RUN.

There are seven leading spaces before each line — this puts the first character of code in column 8, which is what fixed-format COBOL requires. If you accidentally start in column 1, the compiler will read your code as a sequence number and fail.

Compile and run

Open a terminal in the directory containing hello.cob and run:

cobc -x -o hello hello.cob

This produces an executable named hello in the current directory. The flags:

  • -x — build an executable (the default is a loadable module).
  • -o hello — name the output file hello.

Run it:

./hello

(On Windows from a regular cmd or PowerShell prompt, the executable is called hello.exe and you can run it with just hello.) Expected output: 

Hello, World!

If you see that line, your toolchain is working and you are ready to start the course proper.

Try free format

🔍

Most of the example programs later in the course use free-format source — indicated by the >>SOURCE FORMAT IS FREE directive at the top of the file. Free format removes the column 7–72 restriction and lets you start code anywhere on the line.

For a free-format version of the same program, create hello-free.cob:

>>SOURCE FORMAT IS FREE
IDENTIFICATION DIVISION.
PROGRAM-ID. Hello.

PROCEDURE DIVISION.
    DISPLAY "Hello, World!".
    STOP RUN.

And compile the same way:

cobc -x -o hello-free hello-free.cob

Free format is easier to read and edit, especially in a modern editor. The course mixes the two formats to give you exposure to both.

When it doesn't work

Common first-run errors

Here are the errors you are statistically most likely to hit on your first program, and what each one means.

cobc: command not found

The compiler installed, but the directory containing cobc is not on your PATH. Check the install steps above for the right shell to use (MSYS2 UCRT64 on Windows; a fresh terminal session on Linux/macOS to pick up shell init changes).

On Windows specifically: the cobc binary lives inside the MSYS2 install directory and is only on your PATH when you open one of MSYS2's shells. Either use that shell for development, or add the binary directory to your system PATH.

syntax error, unexpected ... on line 1

Almost always a column-alignment problem in fixed-format source. Common causes:

  • Code starts in column 1 instead of column 8. (The first 6 columns are reserved for an optional sequence number and the 7th is a status indicator.)
  • The file was edited in an editor that converted tabs to a different width — something that looks aligned visually may not be aligned in actual columns.

If you don't want to think about columns, add >>SOURCE FORMAT IS FREE as the first line of your source file. With free format the column rules go away.

PROGRAM-ID is required / division order errors

A COBOL source file has a fixed structure: every program must start with the IDENTIFICATION DIVISION and a PROGRAM-ID entry. Even Hello, World is required to declare those two things.

If you see an error about a missing division or about divisions being out of order, the fix is almost always to add the boilerplate at the top of the file shown in the Hello, World section above.

The compile succeeded but there is no output

If you forgot the -x flag, cobc produced a loadable module (hello.so on Linux/macOS, hello.dll on Windows) rather than an executable. Run it with cobcrun hello, or recompile with -x to get a standalone executable.

Numeric errors (later in the course)

🔍

Save these for later, but worth knowing they exist before you hit them.

When you start using arithmetic and numeric edited fields, two errors show up regularly:

  • libcob: numeric field value invalid — a field declared as PIC 9 or similar contains non-numeric data (often spaces, often because the field was never initialised). Initialise with VALUE ZEROS at declaration time and the problem goes away.
  • libcob: subscript out of bounds — an OCCURS reference indexed past the end of the table. Compile with -fdebugging-line and -fbounds-check while developing to surface these earlier.

Beyond GnuCOBOL

If your goal is to work on a real mainframe, two further chapters of this course will be especially relevant once you have the language under your fingers:

  • JCL — the job-control language that wraps batch COBOL programs on z/OS. Without JCL you cannot run COBOL on the mainframe.
  • Embedded SQL / DB2 — the standard way commercial COBOL talks to a relational database.

Neither of those topics is needed for the language tutorials. Tackle them when you have a specific reason to.

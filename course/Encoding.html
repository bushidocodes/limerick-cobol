Introduction

Aims

The course teaches PIC X character data, alphabetic comparisons, and INSPECT / STRING operations without ever mentioning the elephant in the room: mainframe COBOL runs in EBCDIC; everything else runs in ASCII or UTF-8. Two consequences a learner will hit immediately when they leave the classroom:

  1. Collating sequences differ. 'A' < 'a' in ASCII; 'a' < 'A' in EBCDIC. 'A' < '1' in EBCDIC; '1' < 'A' in ASCII. SORT results and condition-name ordering depend on this.
  2. File interchange. A flat file written on a z/OS host and FTP'd in binary mode is unreadable on a Linux box, and vice versa. Packed-decimal fields encode digit nibbles identically in both — but signed-decimal sign nibbles differ.

Modernisation projects routinely founder on this. This unit gives you the model.

Objectives

By the end of this unit you should —

  1. Understand what an encoding is and why two systems with different defaults can disagree about the same byte.

  2. Be able to predict whether a SORT result will be the same on a mainframe and a Linux box, and override it if needed.

  3. Know what binary FTP between EBCDIC and ASCII systems preserves and what it breaks.

  4. Recognise PIC N and USAGE NATIONAL and know what their role is in modern COBOL.

What a character encoding is

Bytes are not characters

On disk and in memory, a string of characters is stored as a sequence of bytes. The encoding is the mapping from byte values to characters: which byte represents capital A, which represents lowercase a, which represents the digit 0.

If you write a file on one machine in one encoding and read it on another machine in a different encoding, the bytes are the same but the characters they represent are different. In the worst case the program runs without error and silently produces wrong output — by the time someone notices, the corruption may already be downstream.

The COBOL platforms a working programmer will meet use two main encodings: EBCDIC on IBM mainframes (z/OS, z/VSE, AS/400), and ASCII — or its UTF-8 superset — on everything else.

EBCDIC vs ASCII at a glance

Selected code points

🔍

The full mapping has 256 entries and several variant code pages (CP037 for US English, CP285 for UK, CP1047 for OS/390 Java, …). The table below shows the printable characters whose ordering matters most.

Character ASCII (hex) EBCDIC CP037 (hex)
space 20 40
'0' (digit zero) 30 F0
'9' 39 F9
'A' 41 C1
'Z' 5A E9
'a' 61 81
'z' 7A A9
'$' 24 5B
'#' 23 7B

The pattern to notice: within each encoding, digits and letters are contiguous and ordered, but the ordering of digits relative to letters and of uppercase relative to lowercase differs.

Collating sequence

The same condition, two different answers

Comparisons between alphanumeric fields use the host's native collating sequence. Consider: 

01  WS-A   PIC X VALUE "A".
01  WS-1   PIC X VALUE "1".

IF WS-A > WS-1
    DISPLAY "Letters sort after digits"
ELSE
    DISPLAY "Digits sort after letters"
END-IF.

On a z/OS host (EBCDIC) the program prints "Digits sort after letters" — the byte for 'A' is C1, the byte for '1' is F1, and 'A' < '1'.

On a Linux host (ASCII) the same program prints "Letters sort after digits" — 'A' = 41, '1' = 31, so '1' < 'A'.

This is exactly the kind of difference that survives unit tests and breaks in production. A SORT on a part number field that mixes letters and digits will reorder records when moved between platforms; a condition name on a two-letter status code will pick a different branch.

A concrete SORT example

*> Input records (in the order they were written):
A100
1234
B200
4567

*> After SORT ASCENDING on a key field, on z/OS:
A100
B200
1234
4567

*> The same SORT on Linux / Windows:
1234
4567
A100
B200

Same program, same data, different output. If a downstream consumer expects a specific order, that's a defect introduced by the platform change alone.

The ALPHABET clause

Forcing a specific collation

COBOL lets you declare a named collating sequence and apply it either to the whole program or to a specific SORT: 

ENVIRONMENT DIVISION.
CONFIGURATION SECTION.
OBJECT-COMPUTER. IBM-Z
    PROGRAM COLLATING SEQUENCE IS ASCII-Order.

SPECIAL-NAMES.
    ALPHABET ASCII-Order IS STANDARD-1.

STANDARD-1 is the COBOL standard's name for ASCII; STANDARD-2 is ISO-7-bit; NATIVE is the host default; EBCDIC is explicit. With the PROGRAM COLLATING SEQUENCE clause, every comparison and every implicit SORT uses the named alphabet regardless of the host.

For a one-off override, attach the alphabet to a specific SORT: 

SORT WorkFile
     ASCENDING KEY WK-PartNumber
     COLLATING SEQUENCE IS ASCII-Order
     USING InputFile
     GIVING OutputFile.

This is the standard tool for taming the "the report is in a different order on the cloud than on the mainframe" surprise. Pick one collation, declare it explicitly, and the platform stops mattering.

Packed-decimal portability

The (mostly) good news

COMP-3 packed-decimal storage is identical on EBCDIC and ASCII platforms in the digit nibbles. A field declared PIC S9(7) COMP-3 with value +1234567 is stored as the byte sequence 01 23 45 6C on both. The first nibble of each byte holds a decimal digit; the final nibble is the sign.

So if you FTP a file of packed-decimal numbers between systems in binary mode, the values come through correctly … almost.

The sign nibble differs

The conventions for the sign-byte nibble are:

Encoding Positive Negative Unsigned
EBCDIC (z/OS) C D F
ASCII (GnuCOBOL, Visual COBOL) C D F

The values happen to match across modern implementations because both follow the IBM convention. Older compilers and some embedded systems use A/B or E/F. If a file is mysteriously failing arithmetic operations on its COMP-3 fields after a platform move, suspect the sign nibble.

Zoned decimal (PIC S9(7) stored as DISPLAY usage with an overpunched sign) is much worse — the sign is encoded into the high nibble of the last digit byte, which differs between EBCDIC and ASCII. Don't FTP zoned-decimal fields between platforms in binary mode without conversion.

File interchange in practice

Decision matrix

File contents Recommended transfer
Pure text (e.g. CSV, fixed-width report) FTP text mode. The gateway translates EBCDIC↔ASCII as it copies.
Pure binary (e.g. zipped data, compiled load module) FTP binary mode. No translation.
Mixed (text + COMP / COMP-3 fields) Binary, then run iconv or a custom convert step on the text fields only. Or design the file as pure text from the start.
Zoned-decimal fields Avoid. If unavoidable, convert with dd conv=ascii / dd conv=ebcdic at the boundary.
Variable-length records with length prefixes Binary; the length prefixes are not text and must not be translated.

The general rule: if any field in a record is a non-text type (binary, packed decimal, zoned decimal), you cannot use text-mode transfer for the file as a whole. Either redesign the file to be pure text, or carry the binary across with a custom conversion step at the boundary.

The iconv command

On Linux/macOS, iconv is the standard tool for converting between code pages: 

iconv -f IBM-037 -t UTF-8  mainframe-report.txt > ascii-report.txt
iconv -f UTF-8   -t IBM-037 ascii-input.txt        > mainframe-input.txt

On z/OS, the equivalent is the iconv Unix System Services command or, in batch, the DFSORT INREC / OUTREC with CONVERT.

Unicode and national data

PIC N and USAGE NATIONAL

COBOL 2002 added a "national" data type intended for Unicode-style characters. The picture character is N; the usage clause is USAGE NATIONAL. 

01  WS-Greeting    PIC N(20) USAGE NATIONAL.

MOVE NX"30533066306B0061" TO WS-Greeting    *> "こんにちは"

Internally, national data is UTF-16 on IBM Enterprise COBOL and on Visual COBOL. Each national character is two bytes, regardless of how many "characters" it represents in grapheme terms.

The honest scope of Unicode support

National data exists, but as of 2026 the experience is uneven:

  • UTF-16 fixed two bytes per code unit. No support for surrogate pairs in most operations — characters outside the Basic Multilingual Plane are truncated or mangled in INSPECT and STRING.
  • UTF-8 has no first-class support. You can store UTF-8 in a PIC X field but the character count and byte count disagree and string operations work on bytes only.
  • JSON / XML parse and generate (see Modern COBOL features) handle UTF-8 internally regardless of the surrounding data type, which is convenient when integrating with web services.

For new code that needs full Unicode handling, the pragmatic approach is to keep the COBOL core working on canonical 7-bit ASCII or single-byte EBCDIC, and to handle Unicode at the edges (web service, file import) with a separate component.

Introduction

Aims

Every program in the rest of this course is taught as a standalone executable: you compile it, you run it, you see its output. On a mainframe that is not how COBOL runs. The program is wrapped in JCL — Job Control Language — that allocates the datasets it reads and writes, sets its parameters, and submits it to the operating system for execution.

JCL is not COBOL, and it is not pretty. But you cannot run COBOL on a z/OS mainframe without it, and a learner who finishes the language chapters and steps onto a real system will hit a wall without at least a working mental model.

This unit gives you that model.

Objectives

By the end of this unit you should —

  1. Recognise the three core statement types (//JOB, //EXEC, //DD) and understand what each one does.

  2. Be able to read a piece of production JCL and explain how each //DD statement maps onto the COBOL SELECT ... ASSIGN clauses in the program.

  3. Be able to recognise the most common JCL errors and z/OS ABEND codes, and know where to look to start diagnosing them.

  4. Understand the role of cataloged procedures and how a shop typically bundles a compile/link/run flow.

Prerequisites

  • The structure of COBOL programs
  • Introduction to Sequential files
  • Helpful but not required: any of the file-handling chapters that follow Sequential Files.

What JCL is

The job-control language

On a z/OS mainframe, programs are executed as jobs — units of work submitted to a subsystem called JES (Job Entry Subsystem; usually JES2, sometimes JES3). A job is described by a small text file written in JCL. The file says:

  • Who is running this job, and what accounting / priority information goes with it.
  • Which programs make up the job, and the order in which they run. Each one is called a step.
  • Which datasets each step reads from and writes to, and how those datasets are allocated (existing? new? temporary? how much space?).

JES reads the JCL, schedules the job, runs each step, and writes the output back to a spool dataset that the user can inspect afterwards. The COBOL program itself does not know about the JCL — it just opens files using the SELECT name, and the operating system uses the JCL to resolve that name to an actual dataset.

What JCL is not

JCL is not a programming language in the usual sense:

  • No expressions, no variables, no arithmetic. (Symbolic parameters exist, but they are string substitution, not computation.)
  • No looping. Every step is listed explicitly in the JCL; if you want a step to run a hundred times, the JCL lists it a hundred times (or you write a program that does the looping internally).
  • No real conditional logic beyond COND= codes that suppress steps based on the return code of earlier steps.

It is much closer in spirit to a shell script that runs ./program input.dat output.dat, except that the dataset allocation is far more elaborate and the syntax is much older.

The three core statement types

Syntax shape

Every JCL statement starts in column 1, begins with two forward slashes, and is followed by an optional name and a statement type: 

//NAME TYPE  parameter,parameter,parameter
//          parameter,parameter   (continuation)

The three types you will see in nearly every job:

Type What it does
//JOB Declares the start of the job. Appears once at the top, names the job, and carries accounting information.
//EXEC Starts a step. Either runs a program directly (EXEC PGM=name) or invokes a cataloged procedure (EXEC PROCNAME).
//DD A Data Definition. Each one names a dataset the current step's program will use, and tells z/OS how to allocate it. One //DD per file the program opens.

There are a handful of other statement types (//PROC to start a procedure, //IF for conditional execution, //SET for symbolic parameters) but you can read most JCL with only the three above.

The //JOB statement

The job statement looks like this:

//PAYROLL  JOB (ACCT123),'JANE SMITH',CLASS=A,MSGCLASS=H,
//             REGION=0M,NOTIFY=&SYSUID

The key parameters:

  • (ACCT123) — positional accounting field. Site-dependent. Often a cost centre or project code.
  • 'JANE SMITH' — programmer name, also positional.
  • CLASS=A — the job class, used to route the job to the right initiator.
  • MSGCLASS=H — the output class for JES messages. H is often a "hold for review" class.
  • REGION=0M — how much memory the job can use. 0M means "as much as the system allows."
  • NOTIFY=&SYSUID — send the submitter a message when the job completes.

The //EXEC statement

An EXEC statement names the program (or procedure) to run for this step.

//STEP1   EXEC PGM=PAYCALC,PARM='RUN-DATE=20260513',REGION=4M

The parameters:

  • PGM=PAYCALC — the program to run. This corresponds to the PROGRAM-ID. PAYCALC. in the COBOL source after it has been compiled and link-edited into a load library.
  • PARM='...' — up to 100 characters of parameter string passed to the program. The COBOL program receives it via PROCEDURE DIVISION USING ... in its LINKAGE SECTION.
  • REGION=4M — step-level override of the job's region.

The alternative form, EXEC PROCNAME, invokes a cataloged procedure — more on these in Part 6.

The //DD statement

Each //DD statement allocates one dataset for the current step. The name after the slashes is the ddname — the symbolic name the COBOL program will use to refer to the dataset. 

//EMPLOYEE DD DSN=PROD.PAYROLL.EMPMAST,DISP=SHR
//OUTPUT   DD DSN=PROD.PAYROLL.RUNLOG(+1),
//             DISP=(NEW,CATLG,DELETE),
//             SPACE=(TRK,(10,5),RLSE),
//             DCB=(RECFM=FB,LRECL=80,BLKSIZE=27920)
//SYSOUT   DD SYSOUT=*

A COBOL program connects to a DD via the SELECT clause: 

SELECT EmpFile  ASSIGN TO EMPLOYEE
                ORGANIZATION IS SEQUENTIAL.
SELECT LogFile  ASSIGN TO OUTPUT
                ORGANIZATION IS SEQUENTIAL.

So the JCL says "the ddname EMPLOYEE refers to dataset PROD.PAYROLL.EMPMAST" and the COBOL says "the file I call EmpFile lives at ddname EMPLOYEE." When the program runs an OPEN INPUT EmpFile, z/OS resolves the chain.

DD statements in depth

DSN= — the dataset name

The dataset name is the mainframe equivalent of a file path. It is dot-separated and at most 44 characters in total, with each segment up to 8 characters. 

DSN=PROD.PAYROLL.EMPMAST
DSN=USER.JSMITH.JOBS(COMPILE)        for a member of a PDS
DSN=&&TEMPWORK                       for a temporary dataset (only lives for the job)

Datasets named in JCL are normally cataloged, so JES knows which volume they live on. Sites use a naming convention so the high-level qualifier identifies the application (PROD, USER, SYS1, …).

DISP= — disposition

The disposition is a three-part tuple controlling the dataset's state at three moments: before the step, after a successful step, and after a failed step. 

DISP=(status,normal-disposition,abnormal-disposition)

Status (what the dataset is before the step starts):

  • NEW — create it.
  • OLD — it exists; lock it for exclusive use.
  • SHR — it exists; share with other jobs.
  • MOD — it may or may not exist; append if it does.

Normal / abnormal disposition (what to do after the step):

  • KEEP — leave it on the volume; uncatalog if it was temporary.
  • CATLG — keep it and add an entry to the catalog.
  • UNCATLG — keep it but remove from the catalog.
  • DELETE — remove it.
  • PASS — keep it allocated for the next step in the same job.

A common pair you will see everywhere:

  • DISP=SHR — read an existing master file.
  • DISP=(NEW,CATLG,DELETE) — create a new output file; catalog it on success; delete it on failure.

SPACE= — allocation

For a NEW dataset you must tell z/OS how much room to set aside. Space is requested in tracks, cylinders, or blocks of a given size. 

SPACE=(TRK,(10,5),RLSE)
        unit  primary,secondary  release-unused-on-close

So that example asks for 10 tracks primary, up to 5 additional tracks per extent if more space is needed, and releases any unused space when the dataset is closed.

DCB= — record characteristics

The Data Control Block subparameters describe how the records inside the dataset look.

DCB=(RECFM=FB,LRECL=80,BLKSIZE=27920)
  • RECFM=FB — fixed-length, blocked. Common alternatives: VB (variable-length, blocked), F (fixed, unblocked), U (undefined / direct-access).
  • LRECL=80 — logical record length in bytes. Must match the COBOL FD definition.
  • BLKSIZE=27920 — the physical block size on disk. Most modern shops set this to 0 and let the system choose an optimum.

SYSOUT= — print and spool

SYSOUT=* sends the dataset to the job's default output class — usually the JES spool, where the submitter can read it after the job ends. This is how COBOL DISPLAY output is captured: the program writes to a ddname called SYSOUT, which the JCL allocates as SYSOUT=*. 

//SYSOUT   DD SYSOUT=*
//SYSPRINT DD SYSOUT=*

A walkthrough job

Run an already-compiled program

Here is a complete, working job that runs an existing program named PAYRPT from a load library, reads an employee master file, and writes a report to the spool. 

//PAYRPT01 JOB (ACCT123),'JANE SMITH',CLASS=A,MSGCLASS=H,
//             NOTIFY=&SYSUID
//*
//* Run the payroll report against this month's employee master.
//*
//RUNRPT   EXEC PGM=PAYRPT,REGION=4M
//STEPLIB  DD DSN=PROD.PAYROLL.LOADLIB,DISP=SHR
//EMPLOYEE DD DSN=PROD.PAYROLL.EMPMAST,DISP=SHR
//REPORT   DD SYSOUT=*
//SYSOUT   DD SYSOUT=*

Reading top to bottom:

  1. The //JOB card names the job PAYRPT01 and identifies the submitter.
  2. Lines starting with //* are comments.
  3. The single step RUNRPT executes program PAYRPT.
  4. STEPLIB tells z/OS where to find the PAYRPT load module — in the load library PROD.PAYROLL.LOADLIB.
  5. EMPLOYEE is the input file; the COBOL program's SELECT EmpFile ASSIGN TO EMPLOYEE binds to this.
  6. REPORT and SYSOUT both go to the spool so the user can read them after the job ends.

Reading JES output

After submitting the job, you will see something like the following in SDSF (the standard JES output viewer): 

JES2  JOB LOG -- SYSTEM PROD -- NODE NYORK1
 ICH70001I JSMITH   LAST ACCESS AT 14:21:35 ON SUNDAY, MAY 13, 2026
 IEF236I ALLOC. FOR PAYRPT01 RUNRPT
 IEF237I 0890 ALLOCATED TO STEPLIB
 IEF237I 0892 ALLOCATED TO EMPLOYEE
 IEF237I JES2 ALLOCATED TO REPORT
 IEF237I JES2 ALLOCATED TO SYSOUT
 IEF142I PAYRPT01 RUNRPT - STEP WAS EXECUTED - COND CODE 0000
 IEF285I   PROD.PAYROLL.EMPMAST                          KEPT
 IEF373I STEP/RUNRPT  /START 2026133.1421
 IEF374I STEP/RUNRPT  /STOP  2026133.1421 CPU    0MIN 00.18SEC SRB

The lines you actually care about as a programmer:

  • COND CODE 0000 — the program ended with return code 0. Anything else is worth investigating.
  • Per-DD allocation lines (IEF237I) — useful when a job fails because a dataset doesn't exist or is locked.
  • CPU / elapsed time — for performance tuning.

Common failures

JCL errors vs. ABENDs vs. return codes

A job can fail in three meaningfully different ways:

  • JCL error — the JCL itself is malformed, or refers to a dataset that doesn't exist. The job never starts running. Look for IEFC* messages in the JES log.
  • ABEND — the program ran and then died abnormally. The system catches the failure, terminates the step, and produces a system completion code like S0C7 or U1234. There is usually a dump.
  • Non-zero return code — the program ran to completion but STOP RUN returned a non-zero value, or the program set RETURN-CODE deliberately. The job logs the code; later steps may run or skip based on COND=.

System completion codes you will hit

Code Meaning Typical cause
S0C7 Data exception Almost always: arithmetic on a numeric field that contains non-numeric data (usually spaces, because the field was never initialised).
S0C4 Storage protection / addressing Reading or writing past the end of a table; following an uninitialised pointer; subscript out of range.
S322 CPU time exceeded Step ran for longer than the time limit specified in the JOB or step. Often an accidental infinite PERFORM loop.
S013 OPEN error Dataset DCB attributes (record format, record length) don't match the COBOL FD. Or the dataset doesn't exist.
S806 Module not found The load module named on EXEC PGM= is not on STEPLIB or the system LNKLST. Check the STEPLIB DD.
U-codes User abend The program called ILBOABN or similar to terminate itself with a chosen code. Look in the program source for the abend reason.

The JCL error message

When the JCL itself is bad you get a focused error message, e.g.:

IEFC605I UNIDENTIFIED OPERATION FIELD
IEFC662I INVALID LABEL
IEFC663I UNBALANCED PARENS

Treat these like compiler syntax errors — the line number is usually accurate, and the message names the specific problem.

Procedures (PROCs)

Why procedures exist

A real compile/link/run flow for a single COBOL program takes three or four steps and a dozen //DD statements. Repeating that JCL in every job would be a maintenance disaster. So shops bundle the compile/link/run JCL into a cataloged procedure — the JCL equivalent of a function.

A typical user-facing job then collapses to just a few lines:

//COMPRUN  JOB (ACCT123),'JANE',CLASS=A,MSGCLASS=H
//STEP1   EXEC IGYWCLG,SOURCE=PAYRPT
//COBOL.SYSIN   DD DSN=USER.JSMITH.SOURCE(PAYRPT),DISP=SHR
//GO.EMPLOYEE   DD DSN=PROD.PAYROLL.EMPMAST,DISP=SHR
//GO.REPORT    DD SYSOUT=*

IGYWCLG is IBM's standard "compile, link, go" procedure for Enterprise COBOL. The procedure's COBOL step takes the source on //COBOL.SYSIN; the LKED step link-edits the result; the GO step runs it, and //GO.EMPLOYEE and //GO.REPORT add DDs to that final step.

Common shop procedures

PROC What it does
IGYWC Compile only.
IGYWCL Compile and link.
IGYWCLG Compile, link, and "go" (run the resulting program).

Most large shops also define their own procedures with site-specific defaults (target load library, default region size, audit DDs). Ask a colleague which procedure you should use.

Modern alternatives to TSO/ISPF

Why this matters

For most of the last fifty years, the way to edit and submit JCL has been TSO/ISPF: a green-screen editor on the mainframe itself. It still works, and lots of shops still use it, but a learner who only knows VS Code has two good alternatives.

VS Code + IBM Z Open Editor

IBM publishes a free extension called IBM Z Open Editor that gives VS Code:

  • Syntax highlighting for JCL, COBOL, PL/I, and HLASM.
  • Outline view of jobs, steps, and DDs.
  • Code completion for JCL keywords and DD parameters.
  • Integration with the Zowe CLI (below) for submitting jobs and editing datasets.

Install from the VS Code marketplace by searching for IBM Z Open Editor. Pair it with the Zowe Explorer extension to browse and edit z/OS datasets from VS Code.

Zowe CLI

Zowe is an open-source project that puts a modern REST API and command-line interface in front of z/OS. With the Zowe CLI installed you can: 

zowe jobs submit local-file payrpt.jcl
zowe jobs list jobs --owner JSMITH
zowe jobs view job-status-by-jobid J1234567
zowe files upload file-to-data-set src.cob "USER.JSMITH.SOURCE(PAYRPT)"

This is the same surface area as TSO commands but scriptable from any operating system. Modern CI pipelines for mainframe code (e.g. GitLab pipelines that compile COBOL on z/OS) are built on Zowe.

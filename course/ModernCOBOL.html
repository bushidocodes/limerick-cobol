Introduction

Aims

The earlier chapters of this course target the 1985 ANSI standard. Three standards revisions later, the language has acquired features that real production code now uses: free-format source, intrinsic JSON and XML parsing, user-defined functions, variable-length strings without ceremony, and a considerably enlarged intrinsic function library.

A learner who reads only the existing chapters will be surprised by modern code at any 21st-century shop. This unit closes that gap.

Objectives

By the end of this unit you should —

  1. Recognise free-format source and know when to enable it.

  2. Be able to declare and call a user-defined function with FUNCTION-ID.

  3. Be able to parse a JSON document into a COBOL record and generate JSON from a record, and do the same for XML.

  4. Understand the role of DYNAMIC LENGTH items and the problem they replace.

  5. Know which compiler supports which standard so that you can size feature requests before writing them.

Free-format source

Why this exists

Fixed-format COBOL reserves columns 1–6 for an optional sequence number, column 7 for a status indicator (* comment, - continuation), columns 8–11 for the "A area" (division/section/paragraph names, level 01s, FDs), and columns 12–72 for the "B area" (everything else). Anything past column 72 is silently ignored.

This made sense in 1959 when source code was punched on cards. In 2026 it is a constant footgun: code that looks fine in your editor doesn't compile because a tab landed in the wrong column.

COBOL 2002 added free-format source: the column rules are gone. Code starts wherever you like; comments start with *> in place of column-7 *; lines can be any length.

Side by side

The same program in both formats:

      *> ----- Fixed format -----
       IDENTIFICATION DIVISION.
       PROGRAM-ID. Greet.

       PROCEDURE DIVISION.
           DISPLAY "Hello, World!"
           STOP RUN.
>>SOURCE FORMAT IS FREE
*> ----- Free format -----
IDENTIFICATION DIVISION.
PROGRAM-ID. Greet.

PROCEDURE DIVISION.
    DISPLAY "Hello, World!"
    STOP RUN.

>>SOURCE FORMAT IS FREE is a compiler directive that flips the parser into free-format mode. Many compilers also have a command-line switch that does the same thing globally (for GnuCOBOL, cobc -free; for IBM Enterprise COBOL, SOURCE(FREE)).

User-defined functions

Declaration and call

COBOL 2002 added FUNCTION-ID: a separately compilable function that takes typed parameters and returns a value. Functions complement (but do not replace) subprograms. 

>>SOURCE FORMAT IS FREE
IDENTIFICATION DIVISION.
FUNCTION-ID. Celsius-To-Fahrenheit.

DATA DIVISION.
LINKAGE SECTION.
01  LK-Celsius           PIC S999V99 COMP-3.
01  LK-Fahrenheit        PIC S999V99 COMP-3.

PROCEDURE DIVISION USING LK-Celsius RETURNING LK-Fahrenheit.
    COMPUTE LK-Fahrenheit = LK-Celsius * 9 / 5 + 32.

END FUNCTION Celsius-To-Fahrenheit.

Caller side:

>>SOURCE FORMAT IS FREE
IDENTIFICATION DIVISION.
PROGRAM-ID. ConvertTemp.

ENVIRONMENT DIVISION.
CONFIGURATION SECTION.
REPOSITORY.
    FUNCTION Celsius-To-Fahrenheit.

DATA DIVISION.
WORKING-STORAGE SECTION.
01  WS-Body-Temp         PIC S999V99 COMP-3 VALUE 37.0.
01  WS-Body-Temp-F       PIC S999V99 COMP-3.

PROCEDURE DIVISION.
    MOVE FUNCTION Celsius-To-Fahrenheit(WS-Body-Temp) TO WS-Body-Temp-F
    DISPLAY "Body temperature in F: " WS-Body-Temp-F
    STOP RUN.

The points to notice:

  • The REPOSITORY paragraph names every user-defined function the program will call. This is COBOL's nearest equivalent to an import statement.
  • FUNCTION Celsius-To-Fahrenheit(WS-Body-Temp) calls the function inline as an expression — no separate CALL statement needed.
  • Functions get type-checked. Mismatched argument types produce compile-time errors, unlike CALL's by-reference passing where you might not notice until runtime.

For a single arithmetic helper, a user-defined function is cleaner than a subprogram. For something that does I/O or holds state across calls, a subprogram is still the right shape.

JSON PARSE and JSON GENERATE

JSON PARSE

COBOL 2023 (and as a vendor extension in IBM Enterprise COBOL 6.x since the mid-2010s) lets you parse JSON directly into a COBOL group item without an external library. Names in the JSON map to field names in the record. 

>>SOURCE FORMAT IS FREE
WORKING-STORAGE SECTION.
01  WS-Json-Doc    PIC X(500).
01  WS-Order.
    05  OrderId        PIC X(10).
    05  CustomerName   PIC X(40).
    05  Total          PIC S9(7)V99 COMP-3.

PROCEDURE DIVISION.
    MOVE '{"OrderId":"PO12345","CustomerName":"Alice","Total":99.50}' TO WS-Json-Doc

    JSON PARSE WS-Json-Doc INTO WS-Order
        ON EXCEPTION
            DISPLAY "JSON parse failed"
        NOT ON EXCEPTION
            DISPLAY "Got order " OrderId " from " CustomerName
                    " total " Total
    END-JSON.

The matching is by name. OrderId in the JSON binds to OrderId in WS-Order; case-insensitive by default. To override the mapping (e.g. for JSON names that aren't valid COBOL identifiers), use NAME OF phrase: 

JSON PARSE WS-Json-Doc INTO WS-Order
    NAME OF OrderId IS "order-id"
            CustomerName IS "customer_name"
END-JSON.

JSON GENERATE

The inverse direction: emit a JSON document from a COBOL group item.

JSON GENERATE WS-Json-Doc FROM WS-Order
    COUNT IN WS-Json-Length
    ON EXCEPTION
        DISPLAY "JSON generate failed"
END-JSON.

DISPLAY WS-Json-Doc(1:WS-Json-Length).

COUNT IN stores the byte count of the generated document, since the receiving field is fixed-size and the actual content is shorter.

XML PARSE and XML GENERATE

The XML equivalents

Symmetric with the JSON verbs. XML PARSE was added in COBOL 2002 (well before JSON); XML GENERATE came in 2014. 

>>SOURCE FORMAT IS FREE
WORKING-STORAGE SECTION.
01  WS-Xml-Doc     PIC X(500).
01  WS-Order.
    05  OrderId        PIC X(10).
    05  CustomerName   PIC X(40).
    05  Total          PIC S9(7)V99 COMP-3.

PROCEDURE DIVISION.
    XML GENERATE WS-Xml-Doc FROM WS-Order
        COUNT IN WS-Xml-Length
        XML-DECLARATION
        TYPE OF OrderId IS ATTRIBUTE
    END-XML.

    DISPLAY WS-Xml-Doc(1:WS-Xml-Length).
    *> Produces:
    *> <?xml version="1.0"?>
    *> <WS-Order OrderId="PO12345">
    *>   <CustomerName>Alice</CustomerName>
    *>   <Total>99.50</Total>
    *> </WS-Order>

XML PARSE is event-driven rather than tree-based: you declare a handler paragraph that the runtime calls back for each open tag, close tag, and character data. That's more verbose than the JSON equivalent — if you have a choice, JSON is friendlier in modern COBOL.

DYNAMIC LENGTH items

The old way

In COBOL 85, variable-length data required OCCURS DEPENDING ON: 

01  WS-Name.
    05  WS-Name-Len     PIC S9(4) COMP.
    05  WS-Name-Chars   PIC X OCCURS 0 TO 200 TIMES
                                     DEPENDING ON WS-Name-Len.

This works but is awkward: every reference is via the subordinate fields, the length lives next to the data, and string operations have to be coded manually around the OCCURS.

The new way

COBOL 2014 added DYNAMIC LENGTH. The item behaves like a normal alphanumeric field but its length grows and shrinks as needed: 

01  WS-Name        PIC X DYNAMIC LENGTH.
01  WS-Greeting    PIC X DYNAMIC LENGTH.

MOVE "Alice"     TO WS-Name
MOVE "Hello, "   TO WS-Greeting
STRING WS-Greeting WS-Name "!" INTO WS-Greeting
DISPLAY WS-Greeting    *> "Hello, Alice!"

The runtime manages the underlying storage; FUNCTION LENGTH returns the current length; MOVE automatically grows or shrinks the receiving item. Use it for any field whose length isn't known at compile time — user-entered names, parsed tokens, JSON values, ….

New intrinsic functions

The headlines

The 85 standard had a small intrinsic library. The 2002 / 2014 / 2023 revisions added a large number of functions that are now in every modern compiler. The ones you will reach for most often:

Function What it does Standard
TRIM Strips leading and/or trailing spaces. 2002
SUBSTITUTE String find-and-replace. 2002
SUBSTITUTE-CASE Case-insensitive find-and-replace. 2002
CONVERT Value conversion between numeric and string forms. 2023
STANDARD-COMPARE Locale-aware string comparison. 2002
BIT-OF / BIT-TO-CHAR Bit-string manipulation. 2014
FORMATTED-CURRENT-DATE ISO 8601 timestamp without manually concatenating CURRENT-DATE components. 2014
FORMATTED-DATE / FORMATTED-TIME Parse / format date and time strings. 2014
HEX-OF / HEX-TO-CHAR Hex encoding helpers. 2014
LOCALE-COMPARE Comparison under a named locale. 2002
LOWER-CASE / UPPER-CASE Case folding. (Existed in 85 but expanded for national/Unicode in 2002.) 85 / 2002

A short example combining a few of them:

>>SOURCE FORMAT IS FREE
WORKING-STORAGE SECTION.
01  WS-In         PIC X(60) VALUE "  Hello, world!  ".
01  WS-Out        PIC X(60).

PROCEDURE DIVISION.
    MOVE FUNCTION SUBSTITUTE-CASE(
             FUNCTION TRIM(WS-In),
             "WORLD", "COBOL")
         TO WS-Out
    DISPLAY WS-Out   *> "Hello, COBOL!"
    STOP RUN.

Older code achieves the same with INSPECT and reference modification (see Inspect and Reference modification). The function-style is shorter, more readable, and easier to chain.

If you are writing portable code across all three, "free format + 2002 intrinsics + UDF + DYNAMIC LENGTH" is the safe baseline. Anything from 2014 onward needs to be checked against your target compiler.

Where to go next

Each major compiler publishes a "What's New" appendix describing exactly which features they implement and at what version. IBM Enterprise COBOL's Language Reference, OpenText's Visual COBOL Reference, and the GnuCOBOL release notes are the authoritative sources.

To see the 85-era equivalents these features replace, the Inspect and Reference modification chapters cover the older idioms.

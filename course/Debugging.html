Introduction

Aims

The rest of this course teaches you how to write COBOL. This unit teaches you how to fix it when it doesn't work. The toolset is different from the modern languages a learner may already know: there is no console.log equivalent that just works, no IDE that magically attaches to a running mainframe job. But there is a workflow that every working COBOL programmer follows, and once you have it, COBOL is no harder to debug than anything else.

Objectives

By the end of this unit you should —

  1. Know which compiler options produce useful listings and what to do with them.

  2. Be able to add DISPLAY-based tracing in a way that scales beyond a handful of statements.

  3. Know the source-level debuggers available for your compiler and how to start one.

  4. Recognise the most common z/OS ABEND codes (S0C7, S0C4, S322, S013, S806) and start diagnosing them.

  5. Be able to take a buggy program, diagnose it, and have a plan for fixing it.

The compiler is your first debugger

Compiler options that produce useful output

Long before any runtime debugger, the compiler can tell you a lot about your program. The relevant options on IBM Enterprise COBOL, with the GnuCOBOL equivalents:

Option What it produces GnuCOBOL equivalent
LIST An assembler listing of the generated code, with COBOL statement numbers in the margin. Implicit in -S (emits assembler).
SOURCE A formatted listing of the source code with sequence numbers. -fsource-listing
MAP A map of every data item, its level, offset, and length. Indispensable for diagnosing record-layout bugs. -fsource-listing plus inspecting the generated header.
XREF A cross-reference: every identifier with all source lines where it is read or written. -fcross-reference
SSRANGE Runtime bounds-checking on every OCCURS reference. Costs a little performance; catches "subscript out of bounds" cleanly. -fbounds-check
FLAG(I) Informational diagnostics in addition to warnings and errors. -W and -Wextra

On z/OS, the listings come back as part of the compile-step output, on a DD named SYSPRINT. Bookmark the MAP section — you will look at it every time a data item appears in the wrong place.

DISPLAY-driven debugging

The COBOL equivalent of printf

When the program runs but produces wrong output, the simplest tool is to add DISPLAY statements at the suspect points and watch the values flow through. 

DISPLAY "DEBUG: Entered Calculate-Totals, count=" WS-Count
DISPLAY "DEBUG: After fetch, EmpId=" WS-EmpId " Salary=" WS-Salary

This is fine for a handful of statements. To keep it manageable in a larger program:

  • Prefix every trace string with a marker (above: DEBUG:). Then you can grep them out of the production output, or remove them all later with one editor pass.
  • Use a gate. Declare an 88-level flag and check it before every DISPLAY: 
    01  WS-Trace-Flag    PIC X VALUE "N".
    88  Trace-On         VALUE "Y".

IF Trace-On
    DISPLAY "DEBUG: Entered Calculate-Totals, count=" WS-Count
END-IF
    … and set WS-Trace-Flag from the PARM string or an environment variable.
  • Display the structured form, not the field. DISPLAY "EmpRec=" EmpRec dumps the whole record's bytes; if the record contains numeric fields the display will be garbled. Reference the individual fields by name and add a separator.
  • Display values just before the operation that uses them. A DISPLAY after a COMPUTE tells you the result; one before tells you the inputs — usually more useful.

DEBUGGING MODE and DEBUG-ITEM

The legacy in-language debugger

COBOL 85 included a built-in debugging mechanism: code in a special DECLARATIVES section, activated by SOURCE-COMPUTER ... WITH DEBUGGING MODE, that the runtime calls back into when chosen paragraphs are entered or chosen data items are modified. The handler reads diagnostic information out of the predefined DEBUG-ITEM record.

The feature exists in every standards-compliant compiler. It is also, in practice, almost never used. The reasons:

  • The mechanism is verbose for what it provides.
  • It was deprecated in COBOL 2002.
  • Modern compilers and shops use source-level debuggers (next section) instead.

You should be able to recognise it in legacy code, but you have no reason to add new uses.

Source-level debuggers

GnuCOBOL on a desktop

Compile with -g to retain debug symbols, then run under gdb or cobcd: 

cobc -x -g -o mybug mybug.cob
gdb ./mybug

(gdb) break MAIN-PARAGRAPH
(gdb) run
(gdb) print ws-count
(gdb) next
(gdb) continue

COBOL identifiers are lowercased on the way through the C compiler, so use ws-count rather than WS-Count in gdb commands. Most other things just work: breakpoints, single-stepping, variable inspection, backtraces.

The VS Code COBOL extension also bundles a debug adapter that drives GnuCOBOL via cobcd — the same capability, with a graphical front end.

IBM Debug Tool / Z Open Debugger

On z/OS, IBM ships a debugger product called IBM Z Open Debugger (formerly Debug Tool). It runs on the mainframe and presents a UI via either an ISPF green screen, a web-based debugger, or VS Code through the IBM Z Open Editor extension.

The COBOL program needs to be compiled with TEST(SOURCE) so the debugger has access to source-line mapping. Once attached, you have breakpoints, watchpoints, variable display, and step-into/over for batch programs.

CEDF for CICS

CICS has its own built-in debugger called CEDF (CICS Execution Diagnostic Facility). It is invoked by typing CEDF on a terminal and then running the transaction you want to trace. CEDF stops before and after every EXEC CICS command, displays the command and its parameters, and lets you inspect and modify the COMMAREA and other state.

CEDF is the default starting point for diagnosing a misbehaving CICS transaction in production. See An introduction to CICS programming in COBOL for the surrounding context.

Reading an ABEND

The codes you will hit most often

A z/OS ABEND is the kernel saying "your program died unexpectedly." Each ABEND has a system completion code starting with S for system or U for user-requested. The codes you will hit most often, repeated from the JCL chapter for completeness:

Code Meaning Most-likely cause in COBOL
S0C7 Data exception Arithmetic on a numeric field that contains non-numeric data. Almost always: a field declared PIC 9 was never initialised and contains spaces.
S0C4 Storage protection / addressing Subscript out of range on an OCCURS. Following an uninitialised pointer. Writing past the end of a record.
S0C1 Operation exception Branched into garbage. Usually a corrupted control flow elsewhere.
S322 CPU time exceeded Infinite PERFORM loop, or a job that needs more time than the JCL allows.
S013 OPEN error Dataset RECFM / LRECL / BLKSIZE doesn't match the FD.
S806 Module not found Load module named on EXEC PGM= isn't on STEPLIB or LNKLST.

The first three things to check

When an ABEND happens, before reading anything fancy:

  1. What's the system completion code? Look it up in the table above — that narrows the cause to a category.
  2. What's the offset shown in the JES log? Compare against the LIST output to find the COBOL statement number.
  3. What did the DISPLAY output show before the ABEND? Usually a clue is in the last few lines.

Working with dumps

SYSUDUMP and SYSABEND

When a step abends, z/OS can produce a dump of the program's memory at the moment of failure. The dump is routed through one of two DDs:

  • SYSUDUMP — a "user" dump, containing the program's working storage and a summary of the program-status word.
  • SYSABEND — a much larger system dump that also includes control blocks, JES queues, and system trace data. Useful for diagnosing system-level interactions; usually overkill for an application bug.

Add one or the other to your JCL so the runtime has somewhere to put the dump:

//SYSUDUMP DD SYSOUT=*

The key technique is pairing the dump with the compiler's LIST output: the dump shows the failed instruction's offset; the listing's "STATEMENT NUMBER vs OFFSET" map turns the offset into a specific COBOL line. The two together tell you exactly which line of source code died.

A worked example

The buggy program

Consider this program:

       IDENTIFICATION DIVISION.
       PROGRAM-ID. AvgPay.

       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  WS-Salaries.
           05  WS-Salary OCCURS 5 TIMES   PIC 9(6)V99.
       01  WS-Idx              PIC S9(4) COMP.
       01  WS-Count            PIC 9(4).
       01  WS-Total            PIC 9(8)V99 VALUE ZERO.
       01  WS-Average          PIC 9(7)V99.

       PROCEDURE DIVISION.
           MOVE 50000.00 TO WS-Salary(1)
           MOVE 60000.00 TO WS-Salary(2)
           MOVE 55000.00 TO WS-Salary(3)
           PERFORM VARYING WS-Idx FROM 1 BY 1 UNTIL WS-Idx > WS-Count
               ADD WS-Salary(WS-Idx) TO WS-Total
           END-PERFORM
           COMPUTE WS-Average = WS-Total / WS-Count
           DISPLAY "Average salary: " WS-Average
           STOP RUN.

Run it. On z/OS you get:

CEE3207S The system detected a data exception (System Completion Code=0C7).
IGZ0163W The runtime environment is being terminated.

Diagnose it

Step 1: classify. S0C7 = data exception, almost certainly uninitialised numeric.

Step 2: find the suspect statement. The only line that does arithmetic with a value that depends on uninitialised input is the loop bound: PERFORM VARYING ... UNTIL WS-Idx > WS-Count. WS-Count is declared PIC 9(4) but never assigned, so it contains spaces — non-numeric.

Step 3: confirm. Add a DISPLAY "WS-Count=" WS-Count immediately before the PERFORM. On a system that survives the bad data long enough to print, you will see spaces; on most systems the DISPLAY itself succeeds because it doesn't do arithmetic.

Step 4: fix. Either give WS-Count an explicit VALUE 3 at declaration, or assign it before the loop (here, set it to 3 to match the three populated WS-Salary values). Better still, derive the count rather than hand-coding it.

The same bug under GnuCOBOL

Running the same program locally under GnuCOBOL:

libcob: numeric field value invalid
libcob: Aborting

Same diagnosis. The error message is more direct than the z/OS S0C7, and the program location is reported as a line number in your source file rather than as a hex offset. GnuCOBOL is by far the friendliest environment for learning to read this class of error.

Modern local workflows

Develop locally, deploy to mainframe

Many shops have moved to a workflow where the developer writes and debugs the program against GnuCOBOL on a desktop, then ships the working source to z/OS for compile, bind, and production deployment. The advantages:

  • Iteration is seconds, not minutes (no JCL submit / wait / read SDSF cycle).
  • Modern editor, modern debugger, modern git workflow.
  • Tests can be run locally in CI alongside any other language's tests.

The constraints:

  • Anything mainframe-specific (CICS, JCL, DB2 bind, EBCDIC quirks) cannot be tested locally. Either mock it out or accept that the last leg of testing is on the mainframe.
  • Compiler-specific extensions don't port either way — stick to the standard so code that runs locally also runs on z/OS.
  • See EBCDIC, ASCII, and code pages for the collating-sequence and packed-decimal gotchas that hide behind this workflow.
